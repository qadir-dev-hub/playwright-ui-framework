Follow-up notes covering everything discussed after the first Playwright learning PDF: framework structure, fixtures, environments, reporting, GitHub setup, CI, and scaling.

Main transition: the conversation moved from “how do I learn Playwright?” into “how do I design a Playwright framework that could grow into a real team project?”

1. Framework Structure From Scratch

The framework stage introduced the idea that the project should optimize for maintainability, reuse, and team readability, not just for writing one passing test.

playwright-ui-framework/
  tests/
  pages/
  fixtures/
  test-data/
  utils/
  config/
  scripts/
  playwright.config.ts
  package.json
  tsconfig.json
  .env
  .gitignore
  README.md

Key mental rule:

  • scenario logic stays in tests/
  • page-specific UI behavior stays in pages/
  • shared runner setup stays in fixtures/
  • config values stay in config/ or environment files
  • cross-cutting helpers stay in utils/

2. Custom Fixtures And Framework Wiring

Custom fixtures were introduced as the Playwright-native way to inject framework objects into tests.

import { test as base } from '@playwright/test';
import { TodoPage } from '../pages/todo-page';

type MyFixtures = {
  todoPage: TodoPage;
};

export const test = base.extend<MyFixtures>({
  todoPage: async ({ page }, use) => {
    const todoPage = new TodoPage(page);
    await use(todoPage);
  },
});

export { expect } from '@playwright/test';

This changed tests from:

const todoPage = new TodoPage(page);

to cleaner framework-style tests:

test('user can add a todo', async ({ todoPage }) => {
  await todoPage.goto();
  await todoPage.addTodo('Learn custom fixtures');
});

3. What Data Seeding Means

Data seeding was defined as creating the data the test needs before the actual UI scenario begins.

  • a user account the UI test will log in with
  • a product that must already exist
  • a record that must appear in search results

A fast pattern is API seeding before the UI assertions:

test('user can view seeded todo', async ({ request, page }) => {
  await request.post('https://example.com/api/todos', {
    data: { title: 'Learn seeding' }
  });

  await page.goto('https://example.com/todos');
  await expect(page.getByText('Learn seeding')).toBeVisible();
});

4. One Main Fixture Entry Point

When the fixture layer grows, internal fixture files can be split by concern, but tests should still ideally import from one central module.

fixtures/
  page-fixtures.ts
  auth-fixtures.ts
  data-fixtures.ts
  test-fixtures.ts

The main entry point combines the pieces:

import { test as base, expect } from '@playwright/test';
import { pageFixtures, type PageFixtures } from './page-fixtures';
import { authFixtures, type AuthFixtures } from './auth-fixtures';

type MyFixtures = PageFixtures & AuthFixtures;

export const test = base.extend<MyFixtures>({
  ...pageFixtures,
  ...authFixtures,
});

export { expect };

So tests keep one clean import:

import { test, expect } from '../fixtures/test-fixtures';

5. Environment Strategy

The framework moved away from hardcoded URLs and credentials toward environment-driven config.

export const env = {
  environment: process.env.ENV ?? 'qa',
  baseUrl: process.env.BASE_URL ?? 'https://qa.example.com',
  loginUser: process.env.LOGIN_USER ?? '',
  loginPassword: process.env.LOGIN_PASSWORD ?? '',
};

Important distinction:

  • .env.qa is a plain text data file
  • config/env.ts is TypeScript that reads from process.env
Key chain:
.env.qa -> dotenv loads it -> process.env -> config/env.ts reads it

Why The CI Bug Happened

A GitHub Actions failure exposed that the env file was being loaded after env.ts had already read process.env. The result was an empty baseURL, which made this fail:

await page.goto('/automation-practice-form');

The fix was to load dotenv first and then use process.env.BASE_URL in the Playwright config.

import { defineConfig, devices } from '@playwright/test';
import dotenv from 'dotenv';
import path from 'path';

const testEnv = process.env.ENV ?? 'qa';
dotenv.config({
  path: path.resolve(__dirname, 'environments', `.env.${testEnv}`),
});

export default defineConfig({
  use: {
    baseURL: process.env.BASE_URL,
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },
});

6. Reporting Strategy

The framework introduced a deliberate artifact strategy rather than relying only on terminal output.

use: {
  trace: 'on-first-retry',
  screenshot: 'only-on-failure',
  video: 'retain-on-failure',
}

Artifact roles:

  • HTML report = result dashboard
  • trace = deep replay and debugging
  • screenshot = quick visual failure state
  • video = full playback
  • terminal logs = first failure explanation

A key learning moment was understanding why traces were not appearing: trace: 'on-first-retry' only records on a retry, and local config originally had retries: 0.

8. Practice Form Page Object

A real page object was built around the DemoQA Practice Form.

export class PracticeFormPage {
  constructor(private readonly page: Page) {}

  async goto() {
    await this.page.goto('/automation-practice-form');
  }

  async fillStudentRegistrationForm(data: PracticeFormData) {
    await this.page.getByPlaceholder('First Name').fill(data.firstName);
    await this.page.getByPlaceholder('Last Name').fill(data.lastName);
    await this.page.getByPlaceholder('name@example.com').fill(data.email);
    await this.page.getByPlaceholder('Mobile Number').fill(data.mobile);
  }

  async submit() {
    await this.page.locator('#submit').click();
  }
}

A useful locator correction was made here: the submission modal title was a div, not a semantic heading, so this was not reliable:

this.page.getByRole('heading', { name: 'Thanks for submitting the form' })

The better locator was:

this.page.locator('#example-modal-sizes-title-lg')

9. First Framework Fixture And Test

The fixture exposed the page object:

import { test as base, expect } from '@playwright/test';
import { PracticeFormPage } from '../pages/practice-form.page';

type FrameworkFixtures = {
  practiceFormPage: PracticeFormPage;
};

export const test = base.extend<FrameworkFixtures>({
  practiceFormPage: async ({ page }, use) => {
    await use(new PracticeFormPage(page));
  },
});

export { expect };

The first test then read like a real framework scenario:

import path from 'path';

11. GitHub Actions And CI

The next stage was CI using GitHub Actions. The basic model discussed was:

name: Playwright Tests

on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repo
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install dependencies
        run: npm ci

      - name: Install Playwright browsers
        run: npx playwright install --with-deps

      - name: Run Playwright tests
        run: npx playwright test

Important learning points:

  • GitHub Actions runs on temporary cloud runners, not on the local machine
  • you can watch workflow logs live in the GitHub Actions tab
  • for public repos, Actions is often effectively free enough for learning use
  • you do not see a live visible browser in GitHub; you inspect artifacts and logs afterward

12. Scalability Discussion

The conversation then moved from “basic CI” to “could this work for 5,000-6,000 tests?”

Main conclusion:

GitHub Actions + Playwright can scale, but not as one giant simple job.
A large suite needs sharding, matrix jobs, report merging, and often stronger runner strategy.

Scalable patterns discussed:

  • split smoke vs regression vs nightly runs
  • use Playwright sharding
  • use GitHub Actions matrix jobs
  • merge reports
  • consider larger or self-hosted runners if runtime/cost becomes too high

Good For GitHub-Hosted

  • small to medium suites
  • smoke suites
  • early framework development

Likely Needed At Enterprise Scale

  • sharding and parallel jobs
  • report merging
  • larger runners or self-hosted runners
  • careful suite design

15. Main Takeaways

  • Playwright frameworks should stay close to Playwright’s strengths instead of wrapping everything
  • custom fixtures are the native Playwright way to expose framework objects
  • environment handling must be wired carefully, especially in CI
  • artifact strategy matters as much as test execution
  • GitHub Actions is a strong starting point, but enterprise scale needs better orchestration patterns
