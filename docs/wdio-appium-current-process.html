WDIO and Appium Android Test Process

Node Dependencies

Yes, they are all Node/npm dependencies, but they play different roles.

appium
The Appium server package. It starts the automation server that talks to Android through drivers.

appium-uiautomator2-driver
The Android automation driver used by Appium. It knows how to automate native Android apps using Google's UiAutomator2 framework.

@wdio/appium-service
A WebdriverIO service package. It starts and stops the Appium server automatically when you run WDIO tests, so you do not have to manually run appium first.

In your setup:

WebdriverIO test
   ↓
@wdio/appium-service starts Appium
   ↓
appium receives WebDriver commands
   ↓
appium-uiautomator2-driver talks to Android emulator/device
   ↓
Native Android app is automated

So yes: all are installed through npm, live under node_modules, and are used by your WDIO mobile test runner.

Current Android Test Process

When you run this from the project root:

npm run test:mobile:android

this is what happens:

  1. npm runs the script from package.json:
wdio run config/wdio.android.conf.ts
  1. WebdriverIO loads wdio.android.conf.ts.

That config tells WDIO:

Which tests to run:
tests/mobile/android/**/*.spec.ts

Which device to use:
Pixel 5 / emulator-5554 / Android 11

Which APK to install/open:
Mobile_apps/Android.SauceLabs.Mobile.Sample.app.2.7.1.apk

Which automation engine:
UiAutomator2

Which Appium server:
127.0.0.1:4723
  1. @wdio/appium-service starts a local Appium server.

You do not need to manually run:

appium

WDIO starts it for you.

  1. Appium receives the session request from WDIO.

Appium looks at the capabilities:

platformName: Android
automationName: UiAutomator2
app: Android.SauceLabs.Mobile.Sample.app.2.7.1.apk
udid: emulator-5554
  1. Appium uses appium-uiautomator2-driver.

That driver connects to the Android emulator through ADB, installs/opens the APK, and prepares the native automation session.

  1. WDIO starts your test file:
tests/mobile/android/smoke/login.spec.ts

The test creates screen objects:

const loginScreen = new LoginScreen();
const productsScreen = new ProductsScreen();
  1. The screen object performs actions using WDIO selectors.

For example, login.screen.ts does:

$('~test-Username').setValue('standard_user')
$('~test-Password').setValue('secret_sauce')
$('~test-LOGIN').click()
  1. After login, products.screen.ts validates:
PRODUCTS
  1. When the test finishes, WDIO closes the Appium session.
  2. @wdio/appium-service stops the Appium server it started.

So the simple version is:

npm script
 → WDIO config
 → WDIO starts Appium
 → Appium starts UiAutomator2 session on emulator
 → test uses screen objects
 → WDIO/Appium automate the app
 → assertion checks PRODUCTS
 → session and server shut down
