Combining Your Playwright Framework With Playwright MCP

A practical guide that starts from scratch and shows how to use Playwright MCP alongside a real Playwright framework, with concrete workflows, setup patterns, and examples based on how your current project is evolving.

Core principle: your framework remains the source of truth for committed automation. Playwright MCP becomes the fast exploration, debugging, and discovery layer that helps you build better framework code.

1. The Big Picture

Your Playwright framework and Playwright MCP are not competing approaches. They solve different problems.

  • Your framework is for committed, reviewable, repeatable automation.
  • Playwright MCP is for fast exploration, interactive debugging, and agent-assisted browser work.

If you use both well, your workflow becomes much stronger:

  • discover pages faster
  • find better locators faster
  • debug failures faster
  • keep the final output clean inside your framework

2. What Your Framework Does

Your framework should continue to own the durable assets of the project:

  • page objects
  • fixtures
  • tests
  • environment files and config
  • storage state setup
  • CI workflows
  • reporting rules

These are the things you want in source control and code review.

3. What Playwright MCP Does

Playwright MCP helps you interact with a live browser through an MCP client. It is ideal for:

  • opening a page and inspecting what is really there
  • testing candidate locators
  • checking if an element is in a frame, popup, or hidden state
  • navigating real flows quickly without writing full test code first
  • understanding the page structure before creating a page object

4. Why Combining Them Is Powerful

A lot of automation pain comes from one of two extremes:

  • either everything is coded too early without enough exploration
  • or exploration happens ad hoc but never becomes maintainable code

Combining MCP with your framework gives you a middle path:

  1. explore quickly with MCP
  2. understand the page and flow
  3. extract the stable parts into page objects and tests
  4. commit only the maintainable result

5. What You Need Before You Start

Before combining the two, make sure you already have:

  • a working Playwright project
  • a stable playwright.config.ts
  • at least one page object
  • at least one fixture file
  • basic environment handling

Your project already has these pieces, so you are in a good position.

6. Step 1: Keep The Framework Healthy First

This is the most important rule before you even touch MCP:

MCP should support a framework that already has clear structure. If the framework itself is messy, MCP will only make the mess faster.

For your project, that means continuing to keep: 

pages/
fixtures/
tests/
config/
environments/
playwright/
docs/

clear and intentional.

7. Step 2: Add Playwright MCP To Your Client

The core Playwright MCP setup is usually done through your MCP client configuration.

For Codex-style usage, the official approach is: 

codex mcp add playwright npx "@playwright/mcp@latest"

Or the equivalent config: 

[mcp_servers.playwright]
command = "npx"
args = ["@playwright/mcp@latest"]

This makes Playwright MCP available to the client so it can open and operate a browser through MCP tools.

8. Step 3: Decide How You Want MCP To Run

Before you use MCP, decide which browser state model is best for the task.

Option A: persistent profile

Best when you want login and cookies to remain available across sessions.

Option B: isolated mode

Best when you want a clean test-like session each time. 

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest",
        "--isolated"
      ]
    }
  }
}

Option C: extension mode

Best when you want MCP to connect to an existing Chrome or Edge browser session.

For framework work, I recommend:

  • persistent mode for general exploratory browsing
  • isolated mode for framework-like debugging and reproducible investigation

9. Step 4: Reuse Framework Auth With MCP

This is one of the best integration points between your framework and MCP.

Your framework already creates a saved session, for example: 

playwright/.auth/student.json

That same file can be reused by MCP in isolated mode. 

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest",
        "--isolated",
        "--storage-state=playwright/.auth/student.json"
      ]
    }
  }
}

This is extremely useful because it means:

  • your tests and your MCP exploration can start from the same authenticated state
  • you do not need to log in manually every time
  • your investigation is closer to the real test environment

Recommended workflow

  1. Run your auth setup test to refresh storage state.
  2. Start MCP in isolated mode using that same file.
  3. Navigate directly to authenticated pages.
  4. Explore or debug without rebuilding the login flow every time.

10. Step 5: Use MCP To Discover Better Locators

One of the most practical MCP uses is locator discovery.

Instead of writing guesses into your page object immediately, you can use MCP to:

  • inspect the accessible tree
  • confirm roles and names
  • check whether a button is really visible or disabled
  • see whether text is in the current page or inside a frame

Example workflow

  1. Open the page in MCP.
  2. Inspect the important interactive elements.
  3. Note the strongest user-facing locators.
  4. Put those into your page object.

For example, MCP might help you discover that this is stronger: 

page.getByRole('button', { name: 'Submit order' })

than: 

page.locator('.btn-primary:nth-child(2)')

11. Step 6: Convert Discovery Into Page Objects

This is the step many people skip, and it is the reason their tooling usage never becomes a maintainable framework.

Once MCP helps you understand a page, move that knowledge into code.

Example

Suppose MCP helps you learn that the best login page signals are:

  • #username
  • #password
  • #submit
  • a success URL containing logged-in-successfully

The correct next step is not “remember this mentally.” The correct next step is: 

pages/login.page.ts

with methods like: 

async login(username: string, password: string) { ... }
async expectLoginSuccess() { ... }

That is how MCP strengthens your framework instead of becoming a side activity.

12. Step 7: Use MCP For Debugging Failed Tests

This is another very strong practical use case.

When a test fails, especially in CI, you can:

  1. read the Playwright failure and trace
  2. use MCP to reproduce the same route and state interactively
  3. inspect whether the element is present, hidden, delayed, framed, or changed
  4. update the framework code accordingly

Typical debug scenarios where MCP helps

  • the locator matched the wrong thing
  • the element lives in a popup or iframe
  • the text changed slightly
  • the page needed authenticated state you forgot to reproduce locally
  • the test is racing the UI and you need to understand the real state transition

13. Step 8: Use MCP For New Feature Exploration

When your team or project gets a new page, MCP can become your first-pass exploration tool.

A strong workflow looks like this:

  1. Open the feature in MCP.
  2. Map the main user path.
  3. Identify form fields, menus, dialogs, and success states.
  4. Write the page object.
  5. Write the test.

This reduces wasted coding time because you learn the page before you commit to abstractions.

14. Step 9: Use MCP With API + UI Workflows

If your framework uses API seeding or API cleanup, MCP becomes even more useful.

Example:

  1. Create a record via API using your framework helper or request fixture.
  2. Open the related UI page in MCP.
  3. Inspect whether the UI reflects the seeded state.
  4. Convert what you learned into a stable test.

This is powerful because it lets you use the API for fast setup and MCP for real-browser inspection before writing the final test code.

15. Step 10: Create A Practical Team Workflow

If this framework eventually supports more than just you, the team workflow should be clear.

Recommended team rule

  • MCP is allowed for exploration, debugging, and locator discovery.
  • Committed automation should still live as Playwright tests, page objects, and fixtures.
  • Any useful discovery should be turned into maintainable code or docs.

Good shared habit

17. Common Recipes

Recipe A: explore a new form page

  1. Open the page with MCP.
  2. Inspect labels, placeholders, and roles.
  3. List candidate locators.
  4. Create or update the form page object.

Recipe B: verify an authenticated page

  1. Refresh storage state through your normal auth setup test.
  2. Run MCP with that storage state file.
  3. Navigate directly to the target route.
  4. Inspect UI behavior and convert findings into assertions.

Recipe C: debug popup or iframe behavior

  1. Use MCP to inspect whether a flow opened a popup or uses an iframe.
  2. Confirm the correct browsing context.
  3. Update page object code to use popup or frame-specific handling.

Recipe D: investigate a CI-only failure

  1. Read the report and trace from CI.
  2. Recreate the route and auth state locally.
  3. Use MCP to inspect the live browser state step by step.
  4. Update the test, locator, or setup logic based on what you found.

18. Common Mistakes

  • Using MCP as a replacement for writing proper page objects.
  • Leaving useful discoveries only in chat history instead of converting them into code.
  • Exploring unauthenticated pages when the real test uses authenticated state.
  • Using persistent browser state carelessly and forgetting it may hide setup problems.
  • Relying on unstable exploratory steps instead of stable locators and assertions.
  • Mixing framework logic and ad hoc investigation logic in the same place.

19. What Mature Usage Looks Like

Mature MCP usage inside a framework looks like this:

  • MCP is available and easy to use.
  • Auth can be reused through the same storage state as the test framework.
  • Exploration and debugging are fast.
  • The final product of that work becomes committed framework code.
  • The team has lightweight docs for repeatable MCP recipes.
Best practical advice for you right now:
Use Playwright MCP to speed up page discovery and debugging in your current framework. Keep all durable automation logic in page objects, fixtures, tests, and auth setup files. That combination gives you speed without sacrificing maintainability.
