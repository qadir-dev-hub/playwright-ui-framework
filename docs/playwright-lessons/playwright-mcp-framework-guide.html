Playwright MCP For Your Framework

A detailed guide to setting up and using Playwright MCP alongside a real Playwright automation framework, with special attention to your current project style and Codex-based workflow.

Main idea: Playwright MCP is best used as an exploration, debugging, and discovery layer that supports your framework. It should help you build better page objects, locators, and tests. It should not replace a maintainable framework.

1. What Playwright MCP Is

Playwright MCP is an MCP server built on top of Playwright. It lets an MCP client talk to a browser through structured tools instead of asking you to write all browser actions manually. One of its key ideas is that it works from structured accessibility information rather than depending only on screenshot-based reasoning.

In practice, that means an agent can open pages, inspect accessible elements, click, type, switch tabs, inspect network requests, capture screenshots, and even save or restore browser state.

2. When MCP Is A Good Fit

MCP is especially useful when you are:

  • exploring a new page quickly
  • discovering good locators
  • debugging a failing flow interactively
  • investigating why a page behaves differently in a real browser
  • mapping an unfamiliar UI before turning it into page object methods
  • running an agentic loop where the browser stays open and context is preserved across steps

MCP is less ideal when you already know exactly what code you want and just need stable committed test files. In that case, normal Playwright test code is usually cleaner and more efficient.

3. How MCP Differs From Normal Playwright Tests

Think of the difference like this:

  • Normal Playwright tests are source code you write, version, review, and run in CI.
  • Playwright MCP is an interactive browser automation interface for an MCP client or agent.

So your framework should still contain:

  • page objects
  • fixtures
  • tests
  • environment handling
  • CI workflows

MCP should help you produce and debug those assets faster, not replace them.

4. Prerequisites

Before setting up Playwright MCP, make sure you have:

  • Node.js 18 or newer
  • an MCP-capable client such as Codex, VS Code, Cursor, Windsurf, Claude Code, or Claude Desktop
  • Playwright familiarity, especially basic browser automation concepts

Since you already have a working Playwright project, you are past the hard part. The main setup task now is client configuration.

5. Core Setup For Most MCP Clients

The standard local configuration pattern is simple: tell the client to launch the Playwright MCP server via npx @playwright/mcp@latest.

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest"
      ]
    }
  }
}

This is the default shape that many clients accept, even though each client stores the config in a different place.

6. Codex Setup

Because you are working in Codex-style tooling, the most relevant setup path is the Codex MCP configuration.

The official Playwright MCP repository documents this Codex CLI command:

codex mcp add playwright npx "@playwright/mcp@latest"

You can also configure it directly in Codex config using:

[mcp_servers.playwright]
command = "npx"
args = ["@playwright/mcp@latest"]

That is the cleanest starting point if you want the Playwright MCP server available to your Codex client without manually retyping the command each time.

Recommended practical setup for you

  1. Keep your framework code in playwright-ui-framework.
  2. Add Playwright MCP to your Codex client using the official config.
  3. Use MCP for discovery and debugging.
  4. Translate useful findings into page objects and tests inside the repo.

7. Understanding MCP Transport Modes

The MCP specification currently defines two standard transport types:

  • stdio
  • streamable HTTP

For local setup, stdio is the normal default. In this mode:

  • the MCP client starts the server as a subprocess
  • the server reads requests from standard input
  • the server writes responses to standard output

This is why a simple local config can just specify a command and args. The client handles the subprocess lifecycle automatically.

When HTTP mode matters

HTTP mode becomes useful when:

  • you want the MCP server running separately from the client process
  • headed browser automation is hard to launch directly from the client environment
  • you want a more service-like setup

8. Profiles: Persistent, Isolated, And Extension

Playwright MCP supports three useful browser state modes.

Persistent profile

This is the default. Cookies and login state are preserved between sessions. This feels closest to using a normal browser profile, and it is very useful for exploratory work.

Good for:

  • keeping yourself logged in while exploring a product
  • repeated manual-style workflows with an agent
  • faster iteration when you do not want to log in repeatedly

Isolated mode

In isolated mode, each session starts fresh. When you close the browser, that state is gone. This is closer to test isolation and is often better when you want predictable clean-session behavior.

Example config pattern:

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest",
        "--isolated"
      ]
    }
  }
}

Extension mode

Extension mode lets Playwright MCP connect to an existing Chrome or Edge browser session through the Playwright MCP Bridge extension. This is useful when you want to reuse a live browser with existing tabs and logged-in state.

9. Storage State With MCP

Storage state is one of the most useful bridges between your code-based framework and MCP-based workflows.

The official Playwright MCP docs and repo describe support for loading storage state in isolated sessions using --storage-state.

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest",
        "--isolated",
        "--storage-state=playwright/.auth/student.json"
      ]
    }
  }
}

This is especially relevant for your project because you already created:

  • tests/auth.setup.spec.ts to save auth state
  • playwright/.auth/student.json as the saved session file

How this helps your framework

You can use MCP with the same saved state your Playwright tests use. That means MCP can start from a known authenticated session instead of forcing you to log in manually every time.

Recommended pattern for you

  1. Create or refresh auth state using your normal Playwright setup test.
  2. Point MCP isolated mode at that same storage state file.
  3. Use MCP to explore authenticated screens and flows.
  4. Convert those discoveries into stable page object methods and tests.

10. Browser And Runtime Options

Playwright MCP supports several useful command-line options that matter in framework work.

Headless mode

Playwright MCP runs headed by default according to the Playwright docs. To run headless, add --headless.

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest",
        "--headless"
      ]
    }
  }
}

Browser selection

You can select a browser with --browser. Supported values in the Playwright docs include chrome, firefox, webkit, and msedge.

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest",
        "--browser=firefox"
      ]
    }
  }
}

Config file

For more advanced setup, Playwright MCP supports a JSON config file passed with --config.

npx @playwright/mcp@latest --config path/to/config.json

This is a good option once your usage becomes more structured and you do not want all options living inline in a client config block.

11. Standalone HTTP Server Mode

When direct client-launched browser sessions are awkward, you can run Playwright MCP as a standalone server.

npx @playwright/mcp@latest --port 8931

Then point the client at the MCP HTTP endpoint:

{
  "mcpServers": {
    "playwright": {
      "url": "http://localhost:8931/mcp"
    }
  }
}

This is useful when:

  • the client environment has trouble launching a headed browser directly
  • you want the server running continuously outside the client lifecycle
  • you want to separate server process management from the MCP client itself

12. Practical MCP Workflows For Your Framework

This is where MCP becomes truly useful in your day-to-day project work.

Workflow A: page discovery

  1. Use MCP to open a new page in the app.
  2. Inspect the accessibility snapshot.
  3. Identify stable roles, labels, text, and test IDs.
  4. Write or update the page object based on what you learned.

Workflow B: debugging a failing test

  1. Take the failing route or page from your normal test run.
  2. Use MCP to reproduce the state interactively.
  3. Inspect whether the element is really visible, enabled, framed, or missing.
  4. Update locators or flow assumptions in the actual test code.

Workflow C: authenticated exploration

  1. Create or refresh storageState in your normal framework.
  2. Start MCP in isolated mode with that stored auth state.
  3. Navigate directly to authenticated pages.
  4. Explore user paths without repeating login.

Workflow D: API and UI investigation together

If you are using API + UI testing in your framework, MCP can help investigate the UI side quickly after API setup creates the required data.

13. How MCP Fits With Page Objects And Fixtures

Here is the most important framework rule:

MCP is for discovery and investigation; page objects and fixtures are for stable framework code.

For example:

  • Use MCP to discover that the best locator is getByRole('button', { name: 'Submit' }).
  • Put that locator into your page object.
  • Use your fixture to inject that page object into tests.

Do not leave important workflows only as MCP chat instructions if they belong in the framework as code.

Good MCP-to-framework loop

1. Explore page with MCP
2. Identify stable locators and state transitions
3. Write page object methods
4. Wire into fixtures if reused
5. Commit test and framework code
6. Keep MCP available for future debugging

15. Security And Safety Guidance

Playwright MCP can open pages, manipulate browsers, and in some modes access file paths or persistent profiles. That means setup choices matter.

Be careful with persistent profiles

Persistent mode is convenient, but it can also retain cookies and offline state longer than you expect. Use it intentionally.

Use isolated mode when you need predictable clean sessions

For framework debugging and test-like behavior, isolated mode is often easier to reason about.

Pay attention to file access options

The Playwright MCP repository documents options such as --allow-unrestricted-file-access. That should be enabled only when you truly need it and understand the implications.

Limit host access where appropriate

The repository also documents options like --allowed-hosts and --allowed-origins. Those are useful when you want to narrow what the browser is allowed to reach.

16. Limits And Tradeoffs

Playwright MCP is powerful, but it is not automatically the best tool for every job.

Where it shines

  • interactive discovery
  • debugging unknown pages
  • agentic browser workflows
  • preserving browser context while reasoning through a flow

Where normal code is better

  • version-controlled regression tests
  • CI pipeline execution
  • repeatable framework assets
  • reviewable business scenarios

The Playwright MCP repo itself also notes a tradeoff between CLI-based automation workflows and MCP-based agentic workflows. In other words, MCP is not the universal answer; it is one good tool in a broader automation toolkit.

17. What A Good Day-To-Day MCP Workflow Looks Like

If I were designing a clean, realistic workflow for you, it would look like this:

  1. Use your normal Playwright framework for committed tests, page objects, auth setup, and CI.
  2. Use Playwright MCP when you need to explore a new page or debug a confusing issue quickly.
  3. If auth is needed, reuse your existing storageState file through MCP isolated mode or use persistent profile mode deliberately.
  4. After the exploration, convert what you learned into framework code.
  5. Document any repeatable MCP recipe that the future-you or your teammates will want again.
Best recommendation for your current project:
Start with Codex MCP setup, use Playwright MCP for page discovery and debugging, and keep your existing storageState-based framework as the source of truth for real tests.
